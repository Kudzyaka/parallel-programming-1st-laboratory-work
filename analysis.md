# Анализ производительности обхода матриц в C

В этом документе мы подробно разберем, как именно ваш код решает поставленную задачу, за что отвечают конкретные строки кода и почему мы получили такие разные результаты времени выполнения при обходе матрицы по строкам и по столбцам.

## 1. Выделение памяти и правильный размер

В задании требовалось подобрать размер квадратной матрицы, который займет 70-80% оперативной памяти и не будет использовать файл подкачки (виртуальную память).

В файле [main.c](file:///d:/1_lab_AG/1_lab_AG/main.c) память выделяется следующим образом:
```c
  // Динамическое выделение памяти для 1D массива, который симулирует 2D матрицу
  // N x N позволяет избежать ошибок сегментации при больших размерах и
  // гарантирует непрерывную память.
  int *a = (int *)malloc(N * N * sizeof(int));
```
Здесь выделяется **цельный блок памяти**, симулирующий двумерную матрицу. Тип `int` занимает 4 байта. С размером $N = 50\,000$, матрица содержит $2.5 \times 10^9$ элементов, что дает ровно **10 Гигабайт** (около 70% от 16 ГБ ОЗУ). Размер считается оптимальным, так как он гарантированно целиком поместится в быструю RAM, но при этом достаточно большой, чтобы выполняться до 10 секунд.

## 2. Измерение времени

> *"Обеспечить измерение и вывод времени выполнения заполнения матрицы (не времени выполнения всей программы, а только времени выполнения циклов заполнения матрицы)."*

Вы выполнили это требование идеально, использовав функцию `omp_get_wtime()`. Обратите внимание, как это реализовано, например, в [fill_rows.c](file:///d:/1_lab_AG/1_lab_AG/fill_rows.c):

```c
        start_time = omp_get_wtime();
        // Внешний цикл - по строкам, внутренний - по столбцам
        for (size_t i = 0; i < N; i++) {
            for (size_t j = 0; j < N; j++) {
                // Обращение a[i][j] как к 1D массиву
                a[i * N + j] = 0;
            }
        }
        end_time = omp_get_wtime();
```
Вызовы `omp_get_wtime()` находятся **исключительно вокруг циклов `for`**. Время, затраченное на выделение памяти `malloc()`, операции ввода-вывода или запуск программы, здесь **не учитывается**. Возвращается строгая разница `end_time - start_time`.

## 3. Обход матрицы: Строки против Столбцов

Это самая важная часть лабораторной работы, которая объясняет колоссальную разницу в результатах.

### Вариант 1: Обход по строкам (Быстрый) - [fill_rows.c](file:///d:/1_lab_AG/1_lab_AG/fill_rows.c)
```c
        for (size_t i = 0; i < N; i++) {
            for (size_t j = 0; j < N; j++) {
                a[i * N + j] = 0;
            }
        }
```
В языке Си двумерные массивы логически хранятся в памяти **линейно (по строкам)**. То есть элементы первой строки лежат друг за другом, затем сразу за ними элементы второй строки и так далее. 

Когда процессор запрашивает из памяти элемент `a[i * N + 0]`, он забирает не один элемент, а целую "линию кэша" (cache line, обычно 64 байта или 16 `int`-значений). Поскольку ваш внутренний цикл идет по `j` (по строке), следующие 15 элементов для `j=1`, `j=2` и т.д. **уже заранее загружены в сверхбыстрый L1 кэш процессора**! 
Это явление называется **Cache Hit**, и благодаря ему данный алгоритм работает очень быстро (около **1.3 секунд** с оптимизацией и заполняя огромную память в 10 ГБ).

### Вариант 2: Обход по столбцам (Медленный) - [fill_cols.c](file:///d:/1_lab_AG/1_lab_AG/fill_cols.c)
```c
        for (size_t i = 0; i < N; i++) {
            for (size_t j = 0; j < N; j++) {
                // Обращение a[j][i]
                a[j * N + i] = 0;
            }
        }
```
Здесь внутренний цикл все так же проходит от 0 до N-1 по переменной `j`. Однако теперь `j` умножается на $N$ ($50\,000$). 
Это означает, что на каждой итерации процессор обращается к элементам, которые физически расположены в оперативной памяти на расстоянии в $50\,000 \times 4 = 200\,000$ байт друг от друга.

Процессор запрашивает "линию кэша" из памяти, берет из нее **только один элемент**, и сразу же эта линия ему больше не нужна. На следующей итерации ему снова приходится лезть в "медленную" оперативную память за совершенно новым участком. Это постоянные промахи мимо кэша — загрузка гигантских объемов памяти вхолостую. Это явление называется **Cache Miss** (или "отвал кэша"), и именно из-за него этот код работает более **8 секунд** (почти в 7 раз медленнее!).

## 4. Различия при заполнении сложными данными

Задание требовало протестировать не только нули, но и последовательные (например, `k++`) и случайные значения.

### Псевдослучайные числа без `rand()`
Заполнение "случайными" значениями реализовано очень грамотно:
```c
a[j * N + i] = (i * 31 + j * 17) % 1000;
```
Если бы вы использовали стандартную функцию `rand()`, она содержит внутренние блокировки (мьютексы) или сложное математическое состояние. Вызов `rand()` миллиарды раз занял бы столько процессорного времени, что полностью "замаскировал" бы разницу между скоростью кэша (Cache Hit/Miss). Матрица в любом случае заполнялась бы ужасно медленно. Ваша формула позволяет процессору просто считать (выполнять математику с регистрами, что очень быстро) и четко обнажает проблему с памятью!

## 5. Роль оптимизации компилятора (-O3)

**Без оптимизаций (`-O0`)** компилятор напрямую переводит код в ассемблер. Время обхода по строкам и столбцам было 3-4 секунды и 9-10 секунд соответственно.

**С максимальными оптимизациями (`-O3`)** компилятор `gcc` применяет ряд ухищрений, главное из которых — **Авто-векторизация (Auto-Vectorization)**. Используя инструкции `SIMD` (например, AVX2 или SSE4), процессор на вашей Windows машине начинает заполнять память не по 1 элементу за раз, а по 4 или 8 элементов одновременно за один такт процессора.
Именно поэтому во время выполнения с флагом `-O3` скорость заполнения **по строкам** сократилась с `~3.5` секунд до фантастических **`~1.3` секунд**. Однако векторизация почти **не помогает при обходе по столбцам**, так как элементы лежат не подряд, и их нельзя "склеить" в вектор. Скорость по столбцам осталась плачевной: `~8 секунд`.
