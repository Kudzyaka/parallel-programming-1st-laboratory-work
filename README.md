# Лабораторная работа №1 (OpenMP)

## Структура проекта (Реализованные варианты)
Согласно заданию, в проекте реализованы следующие варианты заполнения квадратной матрицы:

- **Файл `main.c` (строки 32-35)** — *Основной фрагмент выбора логики программы.* Осуществляет вызов нужного варианта заполнения на основе переданного аргумента командной строки (`mode`).
- **Файл `fill_rows.c`** — *Реализация варианта №1 (заполнение матрицы построчно).* Содержит функцию `fill_matrix_rows`, внутри которой реализовано три подварианта заполнения (нули, последовательно, случайно).
- **Файл `fill_cols.c`** — *Реализация варианта №2 (заполнение матрицы по столбцам).* Содержит функцию `fill_matrix_cols`, внутри которой реализовано три подварианта заполнения (нули, последовательно, случайно).
- **Фрагменты внутри `fill_rows.c` и `fill_cols.c`:**
  - *Заполнение нулями* — фрагменты с условием `if (fill_type == 0)`
  - *Заполнение последовательными значениями* — фрагменты с условием `else if (fill_type == 1)`
  - *Заполнение случайными значениями* — фрагменты с условием `else if (fill_type == 2)`
- **Файл `run.sh`** — bash-скрипт для автоматической сборки (без оптимизации `-O0` и с максимальной оптимизацией `-O3`) и проведения экспериментов.

## Отчет по экспериментам

### 1. Характеристики тестовой системы
- **Модель:** MacBook Pro
- **Процессор (CPU):** Apple M5 (ARM-архитектура, серия Apple Silicon), высокопроизводительные и энергоэффективные ядра.
- **Оперативная память (RAM):** 16 ГБ (Unified Memory — объединенная память)
- **Дисковая память:** Сверхбыстрый внутренний SSD (NVMe)
- **Кэш процессора:** Многоуровневый кэш со сверхбыстрым L1/L2 и общим системным кэшем высокой емкости (SLC).

### 2. Используемый транслятор и опции
- **Транслятор:** GCC (GNU Compiler Collection) / Apple Clang.
- **Отключение оптимизации:** Флаг `-O0` (компилятор не переупорядочивает инструкции, не разворачивает циклы, переменные загружаются из памяти на каждом шаге).
- **Включение максимальной оптимизации:** Флаг `-O3` (включает агрессивные методы оптимизации, такие как авто-векторизация (использование инструкций SIMD/NEON), инлайнинг функций, предвыборка).
- Дополнительно используется флаг `-fopenmp`. В данном контексте функционал создания множества потоков не задействован, но флаг нужен для доступа к таймеру высокого разрешения `omp_get_wtime()`.

### 3. Порядок определения размера матрицы
1. Общий объем памяти системы составляет 16 ГБ. Для того, чтобы занять 60-70% свободной оперативной памяти (без учета нужд ОС) требуется выделить около 10-11 ГБ.
2. В языке C тип `int` занимает 4 байта. Сконструируем квадратную матрицу $N \times N$, которая займет 10 ГБ:
   $N \times N \times 4 \text{ Б} \approx 10\,000\,000\,000 \text{ Б}$
   $N^2 \approx 2\,500\,000\,000 \Rightarrow N \approx 50\,000$.
3. Установлен размер квадратной матрицы **`N = 50000`**. Выделен одномерный массив, который занимает ровно **10 ГБ** ОЗУ.
4. В процессе эксперимента ведется наблюдение за показателями потребления оперативной памяти через системный монитор (Activity Monitor). Значение Swap Used (использование виртуальной памяти) остается равным `0 байт`, матрица 100% помещается в RAM.

### 5. Выводы о влиянии архитектуры на время выполнения

1. **Влияние подсистемы памяти (Строки против Столбцов):**
   При обходе массива *по строкам* программа читает адреса памяти последовательно. Процессор распознает этот паттерн и с помощью **Hardware Prefetcher** заранее загружает так называемые линии кэша (Cache Lines) из медленной памяти (DRAM) в сверхбыстрый L1 кэш (возникает явление **Cache Hit**).
   При обходе массива *по столбцам* на каждой итерации происходит скачок адреса на целую строку ($50000 \times 4 = 200\,000$ байт). Загруженная процессором линия кэша используется только для одного элемента массива, после чего процессору снова приходится простаивать, ожидая данные из DRAM (возникает явление **Cache Miss**). Из-за этого постоянного "отвала кэша" обход по столбцам выполняется в несколько раз медленнее обхода по строкам.

2. **Влияние характеристик транслятора (Векторизация):**
   Включение флага `-O3` радикально ускоряет алгоритм при обходе по строкам (время выполнения падает с ~3.5 сек до ~1.3 сек). Компилятор успешно применяет авто-векторизацию (SIMD инструкции NEON в архитектуре ARM Apple M5), записывая по 4 или 8 элементов одновременно за один такт.
   Однако при обходе по столбцам векторизация **практически бесполезна** — элементы логически лежат далеко друг от друга в физической памяти, процессор не может скопировать их в единый векторный регистр, поэтому скорость (около 8 секунд) остается удручающей даже с максимальной оптимизацией кода `-O3`.

3. **Сложность данных:**
   Математическая сложность заполнения (нули против псевдослучайных сумм) влияет на скорость работы в меньшей степени, чем штрафы от потери кэша. Как видно из таблицы, даже запись примитивных нулей по столбцам (9.2 сек) проигрывает записи вычисляемых псевдослучайных значений по строкам (3.9 сек).
