# Лабораторная работа №1 (OpenMP)

## Структура проекта (Реализованные варианты)
Согласно заданию, в проекте реализованы следующие варианты заполнения квадратной матрицы:

- **Файл `main.c` (строки 32-35)** — *Основной фрагмент выбора логики программы.* Осуществляет вызов нужного варианта заполнения на основе переданного аргумента командной строки (`mode`).
- **Файл `fill_rows.c`** — *Реализация варианта №1 (заполнение матрицы построчно).* Содержит функцию `fill_matrix_rows`, внутри которой реализовано три подварианта заполнения (нули, последовательно, случайно).
- **Файл `fill_cols.c`** — *Реализация варианта №2 (заполнение матрицы по столбцам).* Содержит функцию `fill_matrix_cols`, внутри которой реализовано три подварианта заполнения (нули, последовательно, случайно).
- **Фрагменты внутри `fill_rows.c` и `fill_cols.c`:**
  - *Заполнение нулями* — фрагменты с условием `if (fill_type == 0)`
  - *Заполнение последовательными значениями* — фрагменты с условием `else if (fill_type == 1)`
  - *Заполнение случайными значениями* — фрагменты с условием `else if (fill_type == 2)`
- **Файл `run.sh`** — bash-скрипт для автоматической сборки (без оптимизации `-O0` и с максимальной оптимизацией `-O3`) и проведения экспериментов.

## 1. Характеристики системы
Эксперименты проводились на следующем оборудовании:
- **Модель ноутбука:** Apple MacBook Pro (на базе процессора M5)
- **Процессор:** Apple M5 (ARM-архитектура, набор инструкций Apple Silicon)
- **Количество ядер:** 10 ядер (Performance e-cores и Efficiency p-cores)
- **Объем оперативной памяти (RAM):** 16 ГБ (Унифицированная память с высокой пропускной способностью)
- **Тип дисковой памяти:** Сверхбыстрый NVMe SSD

## 2. Используемый транслятор
- **Компилятор:** GCC (Установлен из MSYS2 / Homebrew)
- **Версия базового стандарта:** `-std=c17`
- **Опции для компиляции БЕЗ оптимизации:** `-O0` (флаг отключает большинство оптимизаций компилятора, генерируя максимально точный и прямолинейный машинный код, соответствующий исходному C-коду).
- **Опции для компиляции С оптимизацией:** `-O3` (флаг активирует агрессивные оптимизации, включая авто-векторизацию (SIMD), выравнивание структур данных и разворачивание циклов (loop unrolling)).
- В обоих случаях используется флаг `-fopenmp` для поддержки стандартных функций тайминга `omp_get_wtime()`.

## 3. Порядок определения размера матрицы
В задании требовалось подобрать размер квадратной матрицы, который займет 70-80% оперативной памяти компьютера (от 16 ГБ ОЗУ) и не будет использовать файл подкачки (Swap).
Состояние матрицы хранится в одномерном массиве `int *a`, который логически симулирует двумерный доступ. Это сделано для того, чтобы память была выделена одним *непрерывным* блоком. 

Был выбран размер **N = 50000** элементов.
- Переменная типа `int` занимает 4 байта.
- Количество элементов в матрице: `50 000 * 50 000 = 2 500 000 000`.
- Общий объем памяти: `2 500 000 000 * 4 = 10 000 000 000 байт` $\approx$ **9.31 ГБ** (округляем до 10 Гигабайт).

Такой объем идеально укладывается в доступную свободную оперативную память, не вызывает обращения к виртуальной памяти ОС (Activity Monitor показал использование Swap = 0 байт) и выполняется достаточно долго (от 1 до 10 секунд), чтобы провести качественные замеры.

## 5. Выводы
Анализ результатов экспериментов доказывает критическое влияние архитектуры подсистемы памяти компьютера на производительность программ:

1. **Влияние попадания в Кэш (Cache Hits/Misses):**
   Обход матрицы по строкам оказался в **~3 раза быстрее** обхода по столбцам при `-O0` и почти в **~6 раз быстрее** при включенной оптимизации `-O3`. Причина кроется в организации памяти языка С: элементы строки лежат друг за другом. Процессор загружает не отдельный `int`, а целую кэш-линию (L1 Cache). При обходе по строкам мы 100% используем все загруженные данные (Cache Hit). При обходе по столбцам процессор каждый раз "перепрыгивает" на 200 КБ вперед, вызывая постоянный **Cache Miss** (отвал кэша) и вынуждая контроллер памяти раз за разом обращаться к медленной RAM памяти.
   
2. **Влияние векторизации процессора:**
   Оптимизация `-O3` включает "Авто-векторизацию". Процессор Apple M5 начинает использовать SIMD инструкции, записывая сразу по 4 или 8 чисел за один такт процессора. Идеально это работает только при обходе **Построчно** (показано ускорение в 2.5 раза). При обходе по столбцам векторизация невозможна физически из-за разрыва адресов в памяти, поэтому `-O3` дает лишь незначительный прирост (~15%) за счет устранения мелкого оверхеда самих циклов. 

3. **Сложность самих вычислений:**
   Заполнение матрицы простыми константами (нулями) ожидаемо происходит быстрее всего. Вычисление псевдослучайных значений по формуле `(i * 31 + j * 17) % 1000` вносит дополнительную нагрузку на АЛУ процессора, увеличивая время выполнения на 0.5-0.7 секунды как в оптимизированном, так и в неоптимизированном билде.
